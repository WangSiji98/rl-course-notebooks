# LLM 里的强化学习总结

这份笔记不是从传统强化学习课程角度来写，而是从“大语言模型训练流程”角度来解释：

- LLM 训练里哪里会用到强化学习
- 为什么会出现 PPO
- 它在做什么
- 和监督微调、偏好学习、DPO 等方法是什么关系

---

## 1. 先说结论

在 LLM 训练里，强化学习通常不是最开始的主体，而是**后训练（post-training）阶段**的一部分。

一个常见的大模型训练流程可以粗略分成：

1. 预训练 `Pretraining`
2. 监督微调 `SFT, Supervised Fine-Tuning`
3. 偏好对齐 / 强化学习阶段

其中：

- 预训练：主要是 next-token prediction，不是强化学习
- SFT：主要是监督学习，不是强化学习
- 强化学习：通常出现在“让模型更符合人类偏好、回答更有帮助、更安全”这一阶段

所以一句话概括：

**LLM 里的强化学习，主要是拿来做对齐，而不是拿来替代预训练。**

---

## 2. 为什么 LLM 需要强化学习

预训练模型学到的是：

- 语言统计规律
- 世界知识
- 推理模式的一部分

但预训练目标本身只是：

**预测下一个 token。**

这会带来一个问题：

- 模型会“像互联网上的文本”
- 但不一定“像一个好助手”

比如仅靠预训练，模型可能：

- 回答很啰嗦
- 不够听指令
- 风格不稳定
- 会生成用户不想要的内容
- 在多个“都语法正确”的回答里，选不到最符合人类偏好的那个

这时候就需要一种方法，让模型学会：

**在很多“都能说通”的回答里，更倾向输出人类更喜欢的那种。**

这就是对齐阶段要解决的问题。

---

## 3. LLM 里“强化学习”到底在优化什么

传统强化学习里，智能体和环境交互：

- 看状态 `state`
- 采取动作 `action`
- 得到奖励 `reward`

在 LLM 里，可以做一个近似类比：

- `state`：当前 prompt 和已经生成的上下文
- `action`：生成下一个 token，或者整段回答
- `reward`：这个回答有多符合人类偏好

所以你可以把 LLM 对齐阶段理解成：

**模型生成回答，然后根据“这个回答好不好”来继续调整策略。**

这里的“策略”其实就是语言模型本身：

`pi_theta(y | x)`

意思是：

- 在输入 `x` 下
- 生成回答 `y` 的概率分布

强化学习阶段要做的，就是调整这个策略，让高质量回答的概率变大。

---

## 4. RLHF 是什么

`RLHF` = `Reinforcement Learning from Human Feedback`

中文一般叫：

**基于人类反馈的强化学习**

它的核心流程通常是：

1. 先做 SFT
2. 收集人类偏好数据
3. 训练奖励模型 `Reward Model`
4. 用强化学习方法优化语言模型

### 4.1 SFT 阶段

先用人工写的高质量回答做监督微调，让模型先学会：

- 基本听指令
- 基本助手风格
- 基本回答格式

这一步很重要，因为如果模型太差，后面的 RL 很难学。

### 4.2 偏好数据

给同一个 prompt 生成多个回答，然后让人类标注：

- A 比 B 好
- 或给排序

例如：

- Prompt: “解释一下黑洞”
- 回答 A：准确、清晰、有结构
- 回答 B：空泛、啰嗦

人类选择 A 胜过 B。

### 4.3 奖励模型

奖励模型的任务是：

**输入 prompt 和回答，输出一个分数，表示这个回答有多好。**

它并不是直接判断真假世界，而是近似“人类偏好打分器”。

所以在 RLHF 里，真正的 reward 往往不是人类实时打的，而是：

**奖励模型给出来的分数。**

### 4.4 用强化学习优化策略模型

然后让语言模型生成回答，再用奖励模型打分，最后通过 RL 更新语言模型。

这里就进入 PPO 这类方法发挥作用的地方。

---

## 5. PPO 在 LLM 里为什么会出现

`PPO` = `Proximal Policy Optimization`

中文一般叫：

**近端策略优化**

它之所以在 LLM 对齐里很有名，是因为它做了两件很重要的事：

1. 能直接优化“回答质量”这种奖励信号
2. 不会让策略一次更新得太猛

第二点特别重要。

因为在 LLM 里，如果你完全只追求奖励模型分数，可能出现：

- 模型为了刷高 reward，学出奇怪模式
- 分布偏离原始语言模型太远
- 语言质量下降
- 出现 reward hacking

PPO 的“保守更新”思想很适合这里。

---

## 6. LLM 里的 PPO 直觉

PPO 在 LLM 里可以粗略理解成：

1. 模型先根据当前策略生成回答
2. 奖励模型给回答打分
3. 计算这个回答相对基线到底好多少
4. 提高高奖励回答的概率
5. 但限制更新幅度，避免模型一下偏太远

所以它不是简单地说：

“哪个回答分高，就无脑往那个方向冲”

而是：

“往高分方向走，但别一步走太猛”

---

## 7. LLM 里的 PPO 和传统 PPO 有什么对应关系

在传统 PPO 里：

- `state`：环境状态
- `action`：动作
- `reward`：奖励

在 LLM 里可以类比成：

- `state`：prompt + 当前生成前缀
- `action`：下一个 token 或整段生成轨迹
- `reward`：回答质量分数

策略仍然是：

- 当前模型输出 token 的概率分布

价值函数仍然是：

- 估计当前生成状态未来还能拿到多少回报

advantage 仍然是：

- 实际回报比当前预期好多少

所以从数学结构上看，LLM 里的 PPO 不是“另一个完全不同的东西”，而是：

**把 PPO 套到语言生成这个特殊序列决策问题上。**

---

## 8. LLM 里的 reward 一般长什么样

在很多 RLHF 管线里，reward 不只是一个单纯的“回答分数”，而常常是几部分组合：

- 奖励模型分数
- KL 惩罚
- 可能还有格式、长度、安全等辅助项

其中最经典的一项是：

### KL 惩罚

这通常是用来限制新策略不要偏离参考模型太远。

你可以把它理解成：

- 想让模型变得更符合偏好
- 但不想让它完全失去原来语言模型的自然性

所以训练目标里经常会出现一种思想：

**奖励高是好事，但偏离原模型太多要扣分。**

这和 PPO 的保守更新思想非常一致。

---

## 9. 为什么 LLM 里的 RL 难

LLM 用 RL 做对齐很强，但也不轻松，难点很多：

### 9.1 奖励不是真实世界奖励

在很多情况下，reward 不是环境里天然给你的，而是：

- 人类偏好
- 奖励模型预测

所以它本身就可能有偏差。

### 9.2 Reward Hacking

模型可能学会：

- 不是真的变好了
- 而是学会“骗过奖励模型”

比如：

- 套模板
- 学会某种表面上高分的风格
- 过度迎合某些打分特征

### 9.3 训练不稳定

LLM 很大，策略更新如果太猛，会造成：

- 语言流畅性下降
- 风格崩坏
- 模型分布偏移太大

### 9.4 样本成本高

生成回答、打分、更新模型都很贵。

所以 LLM 里的 RL，通常会比经典小环境 RL 更强调：

- 稳定性
- 样本效率
- 工程可控性

---

## 10. 为什么后来很多人又喜欢 DPO 一类方法

虽然 PPO 在 RLHF 里非常经典，但后来很多工作发现：

- PPO 管线复杂
- 训练不稳定
- 需要奖励模型
- 工程成本高

所以后来越来越多方法会用：

- `DPO`（Direct Preference Optimization）
- `IPO`
- `ORPO`
- 以及其他“直接偏好优化”方法

这些方法的核心想法通常是：

**尽量绕开完整 RL 管线，直接从偏好数据优化策略。**

你可以把它们理解成：

- 不完全按“先训练奖励模型，再跑 RL”这条路走
- 而是更直接地从偏好对比数据出发做优化

所以今天谈 LLM 对齐时，PPO 很重要，但并不是唯一主角。

---

## 11. LLM 训练里强化学习常见用到的地方

比较常见的场景有：

### 11.1 指令对齐

让模型回答更像助手，风格更符合人类偏好。

### 11.2 安全对齐

让模型更少输出危险、不合规或不符合产品目标的内容。

### 11.3 推理过程优化

有些训练会对：

- 解题步骤
- 思维链
- 工具调用策略

进行奖励建模或强化学习优化。

### 11.4 Agent / Tool Use

当模型不仅生成文字，还要：

- 调工具
- 调 API
- 多步决策

这时 RL 视角会更自然，因为它真的很像一个序列决策过程。

---

## 12. 把你学过的 RL 概念映射到 LLM

如果你已经学过前面的 RL 课程，可以这样类比：

- `policy`
  LLM 的输出分布，也就是下一个 token 的概率分布

- `reward`
  回答质量分数、偏好分数、安全分数等

- `value`
  当前生成状态未来大概还能拿多少回报

- `advantage`
  这次生成比当前预期好多少

- `PPO clipping`
  更新策略，但不要一下子偏离旧模型太远

所以你前面学的 PPO 思想，迁移到 LLM 里时并没有消失，只是对象从“小车推杆子”变成了“生成文本”。

---

## 13. 一个非常粗略的 LLM 后训练流程图

可以把它简单记成：

1. 预训练语言模型
2. 用高质量示例做 SFT
3. 收集偏好数据
4. 训练奖励模型
5. 用 PPO / RLHF 优化策略模型
6. 再做评估、安全检查、迭代

当然，实际工业流程会复杂得多，但这个主线足够帮助理解。

---

## 14. 这份总结最该记住什么

如果只记 5 句话，我建议记这 5 句：

1. LLM 里的强化学习主要出现在后训练对齐阶段，不是预训练主目标。
2. 它的核心任务是让模型更符合人类偏好，而不只是更会预测下一个 token。
3. PPO 在 LLM 里之所以重要，是因为它能优化奖励，同时限制策略更新不要太激进。
4. 奖励往往来自奖励模型，而不是环境天然给的真实 reward。
5. 现代 LLM 对齐里，PPO 很经典，但也有很多方法在尝试用 DPO 等路线替代完整 RLHF。

## 16. 一句话收尾

**LLM 里的强化学习，本质上是在“语言生成”这个动作空间上做策略优化，让模型更倾向输出人类更喜欢、更有帮助、更安全的回答。**
